----------------------------------------------------------------
【このアプリの概要】
----------------------------------------------------------------

美容サロン向けSEO記事を自動生成し、WordPress・Instagram・Threadsに
自動投稿するツール。
Google AI Studio（Cloud Run）上でホスティングして運用している。

主なファイル：
  src/App.tsx  ── フロントエンド全体（UIとすべてのロジック）※非常に大きい
  server.ts    ── バックエンドAPI（WP投稿・SNS投稿・スケジューラ）
  metadata.json── AI Studioのアプリ設定（基本触らない）


----------------------------------------------------------------
【開発・デプロイの手順】
----------------------------------------------------------------

コードはここ（ローカル or Claude Code）で書いて、GitHubにpushすると
AI Studio Buildに自動反映される。

  git add src/App.tsx server.ts
  git commit -m "fix: 変更内容の説明"
  git push origin main

pushしたあと、AI Studioで最新を読み込んで動作確認する。
ローカル（npm run dev）ではAPIキーが必要なため、本番確認はAI Studioでやる。

GitHubリポジトリ：
  https://github.com/ShinobuDodate/salon-seo-automation-engine


----------------------------------------------------------------
【AI Studio の重要な特性（ここが一番大事）】
----------------------------------------------------------------

★ ブラウザ側のGemini SDK呼び出し → 動く
  AI Studio Build環境では、ブラウザ上でGemini SDKを呼ぶと
  AI Studioが自動で認証してくれる。APIキー不要で動く。

★ サーバー側（server.ts）のGemini呼び出し → 動かない
  server.tsからGeminiを呼ぶと「API key not valid」エラーになる。
  理由：AI Studioのシークレット「GEMINI_API_KEY」は表示名であり、
  Gemini REST APIに送れる本物のキーではない。

  ADC（Cloud RunのメタデータサーバーのトークンでGeminiを呼ぶ方法）も
  試したが、generative-languageのスコープが足りず403エラーになる。
  この問題の解決策は現時点で見つかっていない。

【絶対ルール】
  Geminiを使う処理は、必ずブラウザ側（src/App.tsx）に書く。
  server.tsにGemini呼び出しを追加してはいけない。


----------------------------------------------------------------
【使用するモデル名（変更禁止）】
----------------------------------------------------------------

  gemini-3-flash-preview    ── Flashモデル（最新）
  gemini-3.1-pro-preview    ── Proモデル（最新）

AI Studio Build環境では、このgemini-3系モデルを制限なく使える。
これらを使うためにBuild環境で運用している。

  NG例：gemini-2.0-flash、gemini-1.5-proなど他のモデル名に変える
       → APIキーエラーまたは動作不良になる

モデル名を変える場合は必ずユーザーに確認してから変更すること。


----------------------------------------------------------------
【各機能の実装場所（現在の正しい状態）】
----------------------------------------------------------------

■ 記事生成テキスト（generateBlogPost関数）
  → ブラウザ側でGemini SDKを直接呼び出す
  → 以前はサーバー側（/api/generate-article）に移されていたが、
    APIキーエラーで動かなかったため、ブラウザ側に戻した

■ タイトル案生成（generateBatchPosts関数）
  → ブラウザ側でGemini SDKを直接呼び出す

■ ファイル解析（参照資料のアップロード）
  → TXT / MD / CSV：FileReaderで直接テキスト読み込み（Gemini不要）
  → PDF / 画像：ブラウザ側でGemini SDKに inlineData として送信
  → 以前はサーバー側（/api/extract-file-context）経由だったが、
    ボディパース問題＋APIキーエラーで動かなかったためブラウザ側に移した

■ 画像生成（/api/generate-image）
  → server.ts側。同様のAPIキー問題を抱えている可能性あり。

■ WordPress投稿・SNS投稿・スケジューラ
  → server.ts側。問題なし。


----------------------------------------------------------------
【ファイルサイズ制限】
----------------------------------------------------------------

PDF・画像のアップロード上限：50MB

理由：GeminiのinlineData上限が52428800バイト（50MB）。
      超えると「Document size exceeds supported limit」エラーになる。
      さらに100MB超になると、ブラウザがbase64変換の途中でメモリ不足（OOM）
      クラッシュし、エラーメッセージすら出ない。

対策済み：ファイル選択時にfile.sizeを事前チェックして、50MB超は
          読み込まずに即エラーメッセージを表示する。

TXTファイルは直接テキスト読み込みなので上限は特に設けていない。


----------------------------------------------------------------
【エラー表示の仕組み】
----------------------------------------------------------------

エラーを画面に出すには setNotification を使う。

  正しい書き方：
  setNotification({ message: 'エラーメッセージ', type: 'error' });

  間違いやすい書き方（これだけでは画面に何も出ない）：
  setState({ status: 'error', error: 'メッセージ' });
  ※ state.errorはJSXでレンダリングされていないため、見た目に何も変わらない

バッチモード（isBatchMode=true）でgenerateBlogPost内のエラーが起きると、
「if (!isBatchMode)」の条件でエラーが隠されるコードになっていた。
現在は修正済みで、バッチモード中でもエラーが表示されるようになっている。

【過去にやってダメだったこと（繰り返し禁止）】
----------------------------------------------------------------

1. 記事生成をserver.tsに移した
   → サーバー側APIキーエラーで動かない。ブラウザ側に戻すこと。

2. ファイル解析をserver.tsに残した
   → express.raw()のボディパース問題＋APIキーエラーで動かない。

3. ADC認証を試みた
   → generative-languageスコープ不足で403エラー。現時点で解決策なし。

4. モデル名を勝手に変えた（gemini-2.0-flashなど）
   → ユーザーが強く反対。モデル名変更はユーザー確認が必須。

5. 「最新をAI Studioで読み込んでみてください」を何度も繰り返した
   → ユーザーは毎回試している。1か月間解決できていない問題に対して
     同じ言葉を繰り返すのはNG。

6. 元のやり方ができないからと勝手に代替案を提示した
   → ユーザーのルール：「以前できていたことができないからといって、
     他のやり方で代案するのは禁止」

7. revertが中途半端だった
   → 「最初に戻せ」と言われたとき、本当に動いていた状態まで戻すこと。
     commitを一つ一つ確認して、どの時点まで戻すべきか慎重に判断する。


----------------------------------------------------------------
【ユーザーとのやりとりのルール】
----------------------------------------------------------------

- 必ず日本語で応答する
- 専門用語はなるべく使わない
- コード変更前にTypeScriptチェック（npx tsc --noEmit）を実行する
- 変更後も再度チェックして、変更前と結果が同じことを確認してからpush
- モデル名・認証まわりの変更は必ずユーザーに確認を取ってから行う
- 問題が解決しない場合、勝手に代替案を提示しない
- 勝手に修正しない。「何を・なぜ変えるのか」をユーザーに説明し、
  許可を得てから修正を実行する

【共通コンテンツ（定型文）の挿入位置を3つに整理した】
----------------------------------------------------------------

変更前：6箇所（文頭WP・画像前・画像後・文末WP・文末Instagram・文末Threads）
変更後：3箇所に統一

  1. 文頭（共通・全媒体）
     → WordPress・Instagram・Threads すべてに適用
     → WordPress: </h1>タグの直後に挿入（タイトルの次）
     → Instagram: キャプション1行目（フック文）の後に挿入
     → Threads: 本文の前に挿入

  2. 画像前（WordPress・文末）
     → WordPressの本文末尾・画像の直前に挿入
     → SNSには適用しない

  3. 文末（SNSのみ）
     → Instagram・Threads 共通で本文の後に挿入
     → WordPress には適用しない

グローバル設定とサロン別設定の両方で同じ3項目を表示。

【loop_interval_days フィールドは「分」単位（重要・バグ修正）】
----------------------------------------------------------------

DBの loop_interval_days カラムの値は実際には「分」単位。

  10    = 10分
  1440  = 1日
  10080 = 7日
  43200 = 30日
  86400 = 60日

正しい計算式：loop_interval_days * 60 * 1000（ミリ秒変換）

絶対に * 24 * 60 * 60 * 1000 にしない。
以前この変換式に変更してバグを起こした（次回投稿日が10日後になった）。
server.ts と supabase/functions/process-scheduled-posts/index.ts の
両方で同じ計算式を使う。


----------------------------------------------------------------
【投稿先選択モーダル（postingModal）を新設】
----------------------------------------------------------------

「今すぐ投稿」「予約投稿」「ループ予約」ボタンはすべて postingModal を開く。
旧 loopModal を統合してひとつのモーダルに。

  - mode: 'immediate' / 'scheduled' / 'loop' の3モードをタブで切り替え
  - destinations: string[] で投稿先をチェックボックスで選択
      blog / news / instagram / instagram_story / threads
  - ループモード時のみループ設定（間隔・時刻）を表示
  - 投稿先0個の時はボタン無効化

blogSettings.destinations はデフォルト値として使われる。
投稿前にモーダルで記事ごとに上書きできる（overrideDestinations）。
postToBlog と saveToSupabase の両方に overrideDestinations 引数を追加。


----------------------------------------------------------------
【3サイズ画像生成（16:9・1:1・9:16）】
----------------------------------------------------------------

記事生成時に Promise.all で3枚並行生成するよう変更。
使用モデル：gemini-2.5-flash-image

  [imgResp16x9, imgResp1x1, imgResp9x16] = await Promise.all([
    aspectRatio: "16:9",
    aspectRatio: "1:1",
    aspectRatio: "9:16",
  ])

BlogPost型に追加したフィールド：
  imageUrl      ── 16:9（WordPress・Threads用）
  imageUrl1x1   ── 1:1（Instagramフィード優先）
  imageUrl9x16  ── 9:16（Instagramストーリーズ優先）

自前画像アップロードモードは1枚のみ（1x1・9x16はundefined）。

投稿時の画像割り当て：
  Instaフィード  → 1:1優先（uploadedImageUrl1x1 || uploadedImageUrl）
  Stories       → 9:16優先（uploadedImageUrl9x16 || uploadedImageUrl1x1 || uploadedImageUrl）
